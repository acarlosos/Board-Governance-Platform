# Fase 19B.1 — Invalidação inteligente de cache do Executive Dashboard

> Especificação executável. O Executor deve seguir este documento sem precisar do chat.
> Origem arquitectural: blueprint aprovado em [`docs/features/dashboard.md`](../features/dashboard.md) (secção "QA 19A.8 — findings" e roadmap interno). Convenção `docs/execution/` registada em [`.cursor/rules/documentation.mdc`](../../.cursor/rules/documentation.mdc).

## 1. Contexto

- **L1** vive em `App\Services\Dashboard\DashboardMetricsService` — `dashboard_metrics:v1:{cacheSegment}:{period}`, `Cache::remember`, TTL 90 s.
- **L2** vive em `App\Services\Dashboard\Executive\ExecutiveDashboardReadService::sharedKey()` — `dashboard_snapshot:{snapshot_version}:{cacheSegment}:{period}:shared:plain`, `Cache::flexible(60, 120)`.
- **Priorities/Activity** não têm cache partilhado (D3). **Fora desta fase.**
- **`cacheSegment()`** em `App\Services\Reporting\ReportingContext`: `t_{tenantId}` | `global` | `none`.
- **Períodos**: `App\Enums\DashboardMetricsPeriod` — `this_month`, `last_30_days`, `all_time` via `DashboardMetricsPeriod::filterOptions()`.
- **Observers** existentes em `app/Observers/` para todos os 7 modelos relevantes: `TaskObserver`, `MeetingObserver`, `VoteObserver`, `MinuteObserver`, `SignatureRequestObserver`, `SignatureRequestSignerObserver`, `NotificationCenterObserver`. **NÃO criar novos.**
- Driver de cache em produção: `database` (default da Board). **Sem cache tags.**
- Sem queue. Sem events de domínio. Sem endpoint API.

## 2. Decisões finais confirmadas

| Decisão | Conteúdo |
|---|---|
| **D11** — `cacheSegment === 'global'` | **Não invalidado por evento em 19B.1.** TTL natural (L1 90 s / L2 60 s) é aceitável para super_admin global. Razão: invalidar global a cada mutação em qualquer tenant anula o benefício do cache. |
| **D12** — campos KPI por observer | Cada observer mantém `private const KPI_FIELDS = [...]` (lista branca). `updated` só invalida se `array_intersect_key($model->getChanges(), array_flip(self::KPI_FIELDS)) !== []`. `created`/`deleted`/`restored` invalidam incondicionalmente. |
| **§5.5** — `SignatureRequestSignerObserver` | **Não invalidar directamente.** Cascata: signer altera estado do parent `SignatureRequest`, o `SignatureRequestObserver::updated` flui a invalidação. Se em validação surgir um cenário onde o parent não é tocado mas o KPI muda, **parar e abrir issue** — sem improvisar. |
| **Driver de cache** | Não usar **cache tags**; usar `Cache::forget()` explícito por chave. Compatível com `database` driver. |
| **Síncrono** | Invalidação **síncrona** no observer. Sem queue, sem listeners assíncronos. |
| **Activity feed / `AuditLog`** | **Não invalidar** — feed é sempre fresh (D3). |

## 3. Escopo

1. Criar 2 novas classes: `ExecutiveDashboardCacheKeys` (helper de chaves) e `ExecutiveDashboardCacheInvalidator` (serviço de invalidação).
2. Refactorar `ExecutiveDashboardReadService::sharedKey()` e `DashboardMetricsService::getMetrics()` para delegarem a `ExecutiveDashboardCacheKeys`.
3. Estender 6 observers existentes (Task, Meeting, Vote, Minute, SignatureRequest, NotificationCenter) com a invalidação por campos KPI.
4. Adicionar 16 testes (4 unit + 12 feature).
5. Documentar D11, D12 e o novo bloco "Invalidação de cache (19B.1)" em rules e ficha de feature.

## 6. Implementação passo a passo

> Padrão: **3 PRs pequenos**, cada um verde no `php artisan test` antes do próximo abrir.

### Passo 1 — Helper + Invalidator + refactor (PR 1)

1. Criar `App\Services\Dashboard\Executive\Cache\ExecutiveDashboardCacheKeys`:
   - `final class`, métodos estáticos puros.
   - `public static function l1Key(string $cacheSegment, DashboardMetricsPeriod $period): string` → retornar `'dashboard_metrics:v1:'.$cacheSegment.':'.$period->value`. **String exacta** (manter compatibilidade com chaves já em circulação).
   - `public static function l2Key(string $cacheSegment, DashboardMetricsPeriod $period): string` → retornar `'dashboard_snapshot:'.config('board.dashboard.snapshot_version', 'v1').':'.$cacheSegment.':'.$period->value.':shared:plain'`. Idêntica à actual em `ExecutiveDashboardReadService::sharedKey()`.
   - `public static function allKeysForSegment(string $cacheSegment): array` → 6 strings (3 períodos × 2 layers).
   - `public static function periods(): array` → `DashboardMetricsPeriod::filterOptions()`.
2. Criar `App\Services\Dashboard\Executive\Cache\ExecutiveDashboardCacheInvalidator`:
   - `final class`, constructor `__construct(private readonly Illuminate\Cache\Repository $cache)`.
   - `public function invalidateForTenant(int $tenantId): void` → `$this->invalidateForSegment('t_'.$tenantId)`.
   - `private function invalidateForSegment(string $cacheSegment): void` → itera `ExecutiveDashboardCacheKeys::allKeysForSegment(...)` e chama `$this->cache->forget($key)` para cada.
   - **Não logar nada por padrão**. Se necessário em diagnóstico, `Log::debug` apenas com `cache_segment` (string, sem PII).
3. Refactorar `ExecutiveDashboardReadService::sharedKey()`:
   - Manter `private` e a assinatura.
   - Corpo passa a chamar `ExecutiveDashboardCacheKeys::l2Key($ctx->cacheSegment(), $period)`.
   - Não tocar em mais nada do read service.
4. Refactorar `DashboardMetricsService::getMetrics()`:
   - Substituir o `sprintf` por `ExecutiveDashboardCacheKeys::l1Key($ctx->cacheSegment(), $period)`.
   - `CACHE_PREFIX` da classe deixa de ser usado — pode permanecer como constante (deprecated) ou ser removida. **Preferir remover** se nada externo depender dela (verificar com `Grep`).
5. Executar testes 1–8. Verde antes de seguir.

### Passo 2 — Observers (PR 2)

Para cada um dos **6** observers (`Task`, `Meeting`, `Vote`, `Minute`, `SignatureRequest`, `NotificationCenter`):

1. Adicionar `ExecutiveDashboardCacheInvalidator $invalidator` ao construtor **depois** do `AuditLoggerService $audit` existente. Manter `private readonly`.
2. Adicionar `private const KPI_FIELDS = [...]` conforme tabela §9.
3. Adicionar método privado:
   ```
   private function invalidateCache(<Model> $model): void
   {
       $tenantId = $model->tenant_id;
       if ($tenantId === null) {
           return; // no-op — registos órfãos não tocam em cache.
       }
       $this->invalidator->invalidateForTenant((int) $tenantId);
   }
   ```
4. Chamar `$this->invalidateCache($model)` no fim de:
   - `created($model)` — incondicional.
   - `deleted($model)` — incondicional.
   - `restored($model)` — incondicional (se o observer já tiver este hook; se não, **adicionar** seguindo o padrão dos restantes).
   - `updated($model)` — **só** se `array_intersect_key($model->getChanges(), array_flip(self::KPI_FIELDS)) !== []`.
5. Não invocar a invalidação **antes** das chamadas a `AuditLoggerService` — invalidar é o último passo do hook (consistência: audit primeiro, cache forget depois).
6. **Não tocar** em `SignatureRequestSignerObserver` (§5.5).
7. Executar testes 9–13 + 15. Verde antes de seguir.

## 7. Regras de segurança / tenancy

- **`ExecutiveDashboardCacheInvalidator::invalidateForTenant(int $tenantId)`** — única API pública. Não aceita `User`, não aceita `Model`. Tipo `int` força call-sites a serem explícitos.
- **Sem `auth()` / `request()` / `session()`** dentro do invalidator ou helper de chaves. Regras `dashboard.mdc`/`livewire.mdc` aplicáveis.
- **Sem `Cache::forget('dashboard_metrics:...')` ou `'dashboard_snapshot:...'` em qualquer outro local** — só via invalidator. Se aparecer noutro lado, falha de revisão.
- **`tenant_id === null`** → no-op silencioso. **Nunca** invalidar `global` por causa de um registo órfão.
- **`global`** → **nunca** invalidado em 19B.1 (D11). Mesmo bulk seeders devem respeitar — recomendado correr com `Cache::flush()` quando se faz seed massivo de dados de produção.
- **`none`** → não há cache para invalidar (early-return no read service).
- **Logs**: se necessário diagnosticar, `Log::debug('dashboard.cache.invalidate', ['cache_segment' => $segment])` — **sem** nomes, e-mails, payloads de DTO.
- **Tests obrigatórios** (`tests.mdc`): cobrir cross-tenant + super_admin global + segment 'none' + filtro de campos.

## 8. Estratégia de chaves

### Composição

```
L1: dashboard_metrics:v1:{cacheSegment}:{period}
L2: dashboard_snapshot:{snapshotVersion}:{cacheSegment}:{period}:shared:plain
```

- `cacheSegment` ∈ `t_{id}` | `global` | `none`.
- `period` ∈ `this_month` | `last_30_days` | `all_time`.
- `snapshotVersion` = `config('board.dashboard.snapshot_version', 'v1')`.

### Exemplo para `invalidateForTenant(42)`

Chaves esquecidas (exactamente 6):

```
dashboard_metrics:v1:t_42:this_month
dashboard_metrics:v1:t_42:last_30_days
dashboard_metrics:v1:t_42:all_time
dashboard_snapshot:v1:t_42:this_month:shared:plain
dashboard_snapshot:v1:t_42:last_30_days:shared:plain
dashboard_snapshot:v1:t_42:all_time:shared:plain
```

`t_43`, `global`, `none` ficam **intactos**.

### Como não esquecer um período novo

`ExecutiveDashboardCacheKeys::periods()` delega a `DashboardMetricsPeriod::filterOptions()`. Adicionar um caso novo ao enum propaga automaticamente. Teste 4 (`count() === 3`) é sentinela: falha avisa que enum mudou e nada mais precisou de ser actualizado no helper.

## 9. Observers a alterar

Tabela canónica de campos que disparam invalidação no hook `updated`:

| Observer | Model | `KPI_FIELDS` (constante na classe) |
|---|---|---|
| `TaskObserver` | `Task` | `status`, `due_date`, `completed_at`, `deleted_at`, `tenant_id` |
| `MeetingObserver` | `Meeting` | `status`, `scheduled_at`, `deleted_at`, `tenant_id` |
| `VoteObserver` | `Vote` | `status`, `deleted_at`, `tenant_id` |
| `MinuteObserver` | `Minute` | `status`, `deleted_at`, `tenant_id` |
| `SignatureRequestObserver` | `SignatureRequest` | `status`, `deleted_at`, `tenant_id` |
| `NotificationCenterObserver` | `NotificationCenter` | `status`, `read_at`, `deleted_at`, `tenant_id` |

### Hooks por observer

| Hook | Invalidação |
|---|---|
| `created($model)` | sempre |
| `updated($model)` | só se intersect com `KPI_FIELDS` ≠ ∅ |
| `deleted($model)` | sempre |
| `restored($model)` | sempre (adicionar se não existir) |
| `creating`/`updating`/`saving`/`deleting`/`saved` | **não usar** |
| `forceDeleted` | seguir mesma regra do `deleted` (sempre) — adicionar se observer já tratar |

### Não tocar

- `SignatureRequestSignerObserver` — cascata pelo parent (§5.5).
- `AuditLogObserver` (se existir) — Activity feed é sempre fresh.
- Outros observers em `app/Observers/` que não estão na lista.

## 10. Testes obrigatórios

> Runner: PHPUnit (alinhado com a suite actual). Driver de cache: `array` (default de testes).

### Unit (4 testes, sem DB)

1. `l1Key('t_42', DashboardMetricsPeriod::ThisMonth)` retorna **exactamente** `dashboard_metrics:v1:t_42:this_month`.
2. `l2Key('t_42', DashboardMetricsPeriod::ThisMonth)` retorna **exactamente** `dashboard_snapshot:v1:t_42:this_month:shared:plain` (assumindo `snapshot_version=v1` no config de teste).
3. `allKeysForSegment('t_42')` retorna 6 strings, todas distintas, ordem `[L1×3 períodos, L2×3 períodos]` (ou outra ordem estável — teste assert via `sort()` + comparação).
4. `periods()` retorna 3 casos: `[ThisMonth, Last30Days, AllTime]`.

### Feature — invalidator (4 testes)

5. `invalidateForTenant(1)` esquece exactamente as 6 chaves de `t_1`; chaves de `t_2`, `global` e `none` permanecem intactas. Setup: popular as 12 chaves com valores distintos via `Cache::put`; assertar com `Cache::get`.
6. `invalidateForTenant(1)` **não** dispara invalidação em `global` (D11) — sentinel reforçado.
7. Cache miss observável após invalidação: pré-popular L1 de `t_1` (via `DashboardMetricsService::getMetrics`); invalidar; segunda chamada do `getMetrics` faz queries SQL (medir via `DB::listen`).
8. Idem para L2 (`ExecutiveDashboardReadService::read`): primeira chamada após invalidação recalcula providers Hero/Operations; segunda chamada serve do cache.

### Feature — observers (6 ficheiros, 1–3 testes cada)

9. `Task::factory()->create(['tenant_id' => 1])` invalida chaves de `t_1`. Espião: `Cache::put('dashboard_metrics:v1:t_1:this_month', 'X')` antes; verificar `Cache::get(...) === null` depois.
10. Após criação, `Task::query()->update(['title' => 'x'])` em registo do mesmo tenant **não** dispara nova invalidação (`title` não é KPI field). Pré-condição: re-popular cache de `t_1` entre os dois actos.
11. `Task->update(['status' => Completed])` dispara invalidação.
12. **Replicar 9–11** para `Meeting` (campos `status`, `scheduled_at`), `Vote` (`status`), `Minute` (`status`), `SignatureRequest` (`status`), `NotificationCenter` (`status` e/ou `read_at`).
13. **`SignatureRequestSigner` mutação isolada NÃO invalida** o cache do tenant directamente (espião antes/depois — chaves permanecem). Cobre §5.5.
14. (vazio — fundido em 12)
15. `AuditLog::query()->create([...])` (ou criação real de auditLog via `AuditLoggerService`) **não** invalida cache de dashboard (Activity feed é fresh).

### Feature — anti-leak composto (1 teste)

16. Cenário: 2 tenants A e B. `DashboardMetricsService` pré-popula KPIs de ambos (cache hit). Mutação em A (criar `Task` em A) → cache de A é forget (next `getMetrics(A)` recalcula); cache de B permanece (next `getMetrics(B)` é hit sem queries). Provar com `DB::listen`.
