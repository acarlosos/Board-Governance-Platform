# Resultado — Fase 19B.4 (API `GET /dashboard/snapshot`)

Documento de fecho da execução de [`19B.4-api-dashboard-snapshot.md`](./19B.4-api-dashboard-snapshot.md). Data de validação: 2026-05-11.

## 1. Resumo da implementação

Foi exposto o snapshot executivo via **`GET /api/v1/dashboard/snapshot`** com `SnapshotIndexRequest` (query `period`), `DashboardController::snapshot`, `DashboardSnapshotApiResource` (delega em `ExecutiveDashboardSnapshot::toArray()`), middleware **`abilities:reports:read`** e **`Gate::authorize('view_executive_dashboard')`**, rate limit **60/min** por utilizador. O `bootstrap/app.php` passa a distinguir, em respostas API JSON, **`forbidden_ability`** (token sem scope — `MissingAbilityException` via `getPrevious()` após `prepareException`) e **`forbidden_policy`** (gate/policy); `401` API usa código **`unauthenticated`**. OpenAPI (`docs/openapi/v1.yaml`), `docs/features/api.md` e `docs/features/dashboard.md` foram actualizados. **10 testes** feature novos.

## 4. Decisões relevantes tomadas durante execução

- **D34 — Códigos 403 na API:** O `Handler::prepareException` converte `AuthorizationException` (incl. `MissingAbilityException`) em `AccessDeniedHttpException`. Os callbacks tipados com `AuthorizationException` não eram invocados após `prepareException`. A distinção **`forbidden_ability`** vs **`forbidden_policy`** ficou no render de **`HttpExceptionInterface`** para `403`, inspeccionando `$e->getPrevious()`.
- **D35 — `401` unificado:** `AuthenticationException` em rotas `api/*` passa a responder com `error.code: unauthenticated` (alinha à tabela da spec 19B.4 e ao teste do endpoint).
