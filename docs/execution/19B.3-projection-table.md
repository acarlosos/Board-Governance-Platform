# Fase 19B.3 — Projection table `tenant_dashboard_snapshots`

> Especificação executável. O Executor deve seguir este documento sem precisar do chat.
> Origem arquitectural: [`docs/features/dashboard.md`](../features/dashboard.md) → "Próximos blocos 19B" (entrada **19B.2 → reordenada para 19B.3** após inserção de observability como 19B.2). Convenção em [`.cursor/rules/documentation.mdc`](../../.cursor/rules/documentation.mdc).

## 1. Contexto

A Fase 19B.1 fechou invalidação síncrona de **L1** (`dashboard_metrics`) e **L2** (`dashboard_snapshot:…:shared:plain`). A **19B.2 (concluída)** instrumenta `hit/miss/invalidations` para suportar decisões futuras.

Após **cache miss** (L1 expirado ou invalidado), `ExecutiveDashboardReadService::loadOrComputeShared` **recalcula** Hero + Operations através dos providers — agregações SQL sobre tasks/meetings/votes/minutes/signatures. Para tenants com grandes volumes (≥ 50 k tasks, 10 k+ meetings) esse recálculo pode pesar 200–600 ms em cold start. A projection table persiste o resultado pré-calculado e transforma a leitura num **único `SELECT` indexado** (5–15 ms).

A 19B.2 dará dados para confirmar/refutar a necessidade. Esta spec **pode ser pausada** se o `hit_ratio` L2 em produção ficar ≥ 0.90 e o p95 de leitura ≤ 100 ms — nesse caso, a projection é over-engineering.

## 2. Decisões finais confirmadas

| Decisão | Conteúdo |
|---|---|
| **D19** — Escopo de payload | Projection guarda apenas **Hero + Operations** (mesmos blocos do L2). Priorities/Activity continuam sempre frescos (consistente com D3). |
| **D20** — Modelo de refresh híbrido | (a) Job recurring via Scheduler a cada **5 min** processa tenants marcados `is_stale=true` + tenants sem refresh há > 10 min. (b) `ExecutiveDashboardCacheInvalidator` (19B.1) ganha `markProjectionStale($tenantId)` chamado após `forget`. Job é a **única** fonte de UPDATE/INSERT da projection. |
| **D21** — Fluxo de leitura | `ExecutiveDashboardReadService::loadOrComputeShared` consulta projection **primeiro**; se válida (`is_stale=false` AND `refreshed_at >= now - 10min`), hidrata DTOs a partir dela e **skip** L2 flexible. Se inválida ou ausente, fallback ao comportamento actual (L2 flexible). |
| **D22** — Tenancy | Projection apenas para `cacheSegment === 't_{id}'`. **Sem** projection para `global` (super_admin agrega on-the-fly; tráfego baixo) ou `none` (early-return continua). |
| **D23** — Feature flag | `board.dashboard.use_projection` (env `BGP_DASHBOARD_USE_PROJECTION`, default `false`). Quando `false`: read service ignora a tabela; mantém-se o fluxo actual (L2 `flexible` + observabilidade 19B.2). Job continua a popular **mesmo com flag off**, para validação em staging sem impacto no fluxo de leitura. |
| **D24** — Validade da projection | `is_stale=false` AND `refreshed_at >= now - 10 min`. Caso contrário, fallback. Justificativa: refresh a cada 5 min cobre o caso saudável; 10 min de margem para job atrasado por congestion. |
| **D25** — Sem materialised view DB nativa | `tenant_dashboard_snapshots` é tabela **normal**, gerida por Eloquent. Suporta os 3 drivers do projecto (MySQL, PostgreSQL, SQLite). Materialised view nativa fica como evolução remota (≥ Fase 20). |

## 3. Escopo

1. Migration `tenant_dashboard_snapshots` (5 colunas + 2 índices).
2. Model `TenantDashboardSnapshot` com `BelongsToTenant`.
3. Service `DashboardProjectionService` (`refreshFor`, `findValid`, `markStale`).
4. Job queueable `RefreshTenantDashboardSnapshotJob` (1 tenant × 1 período).
5. Command `dashboard:refresh-projections [--tenant=ID] [--force]`.
6. Scheduled task em `app/Console/Kernel.php`: a cada 5 min.
7. `ExecutiveDashboardCacheInvalidator::invalidateForTenant` chama `markStale`.
8. `ExecutiveDashboardReadService::loadOrComputeShared` consulta projection antes de L2 (quando flag on).
9. Feature flag `board.dashboard.use_projection`.
10. ~13 testes (unit + feature + cross-tenant).

## 4. Fora do escopo

- ❌ Projection para Priorities/Activity (continuam sempre fresh — D3).
- ❌ Projection para `cacheSegment === 'global'` ou `'none'` (D22).
- ❌ Materialised view DB nativa (D25).
- ❌ Backfill massivo histórico — projection cresce on-demand via job.
- ❌ Comando `dashboard:rebuild-all` (sem caso de uso real; uso normal é `--tenant=`).
- ❌ Endpoint API (entra em 19B.4).
- ❌ Remoção de L1/L2 — ambas camadas permanecem (projection é L3 opcional).
- ❌ Alteração de TTLs de L1/L2 (mantidos: L1 90s, L2 60s stale / 120s expire).
- ❌ Alteração do DTO `ExecutiveDashboardSnapshot` shape (D8 preservada).
- ❌ Invalidação por evento de Priorities/Activity.

## 6. Implementação passo a passo

> Padrão: **2 PRs** — (A) tabela + service + job + command, sem ligar ao read service; (B) ligação ao read service via flag + invalidador.

### Passo 1 — Tabela e modelo (PR A)

1. Criar migration com colunas:
   - `id` PK
   - `tenant_id` BIGINT NOT NULL (FK → `tenants.id` ON DELETE CASCADE)
   - `period` STRING(16) NOT NULL — valor do enum `DashboardMetricsPeriod` (`this_month` / `last_30_days` / `all_time`)
   - `payload` JSON NOT NULL — shape `{ hero: {...}, operations: {...}, version: 'v1' }`
   - `is_stale` BOOLEAN NOT NULL DEFAULT false
   - `refreshed_at` TIMESTAMP NULLABLE
   - `created_at` / `updated_at`
2. Índices:
   - UNIQUE(`tenant_id`, `period`) — chave natural
   - INDEX(`is_stale`, `refreshed_at`) — query do job (filtrar stale + ordenar antiga)
3. Criar `App\Models\TenantDashboardSnapshot`:
   - `final class`, traits `BelongsToTenant` + `HasFactory`.
   - `$casts = ['payload' => 'array', 'is_stale' => 'boolean', 'refreshed_at' => 'datetime']`.
   - Scope `scopeValid(Builder $q): void` → `is_stale=false` AND `refreshed_at >= now()->subMinutes(10)`.

### Passo 2 — Service, Job, Command (PR A)

1. `DashboardProjectionService`:
   - `__construct(HeroProvider $hero, OperationsProvider $ops)`.
   - `refreshFor(int $tenantId, DashboardMetricsPeriod $period): void` → constrói `User` representativo do tenant (qualquer `User` activo); chama `$hero->build($user, $period)->toArray()` + `$ops->build(...)->toArray()`; `TenantDashboardSnapshot::updateOrCreate(['tenant_id' => …, 'period' => …], ['payload' => …, 'is_stale' => false, 'refreshed_at' => now()])`.
   - `findValid(int $tenantId, DashboardMetricsPeriod $period): ?TenantDashboardSnapshot` → `TenantDashboardSnapshot::where(...)->valid()->first()`.
   - `markStale(int $tenantId): void` → `TenantDashboardSnapshot::where('tenant_id', $tenantId)->update(['is_stale' => true])`.
   - **Sem** `auth()`, `request()`, `session()`.
2. `RefreshTenantDashboardSnapshotJob implements ShouldQueue`:
   - Constructor: `int $tenantId, DashboardMetricsPeriod $period`.
   - `handle(DashboardProjectionService $svc)` → `$svc->refreshFor(...)`.
   - `tries = 3`, `backoff = [10, 30]`.
3. `RefreshProjectionsCommand`:
   - Signature: `dashboard:refresh-projections {--tenant= : ID específico} {--force : refrescar mesmo válidos}`.
   - `handle()`:
     - Se `--tenant=N`: 1 dispatch por período.
     - Senão: query tenants com snapshots `stale` OU `refreshed_at < now()-10min` OU nunca refrescados (LEFT JOIN ausência), e dispatch para cada combinação tenant×período.
     - `--force` ignora filtros (refresca tudo).
     - Logar via `$this->info(...)` quantidade total dispatched.

### Passo 3 — Schedule (PR A)

1. `app/Console/Kernel.php` — método `schedule`:
   ```php
   $schedule->command('dashboard:refresh-projections')
       ->everyFiveMinutes()
       ->withoutOverlapping(10)
       ->runInBackground();
   ```
2. Test smoke local: `php artisan schedule:list` deve mostrar o entry.

### Passo 4 — Integração read + invalidator (PR B)

1. `ExecutiveDashboardCacheInvalidator`:
   - Adicionar `DashboardProjectionService $projection` ao constructor.
   - Após o loop de `forget` em `invalidateForSegment`, chamar `$this->projection->markStale($tenantId)` (extrair `$tenantId` de `$cacheSegment` removendo prefixo `t_`).
2. `ExecutiveDashboardReadService::loadOrComputeShared`:
   - Adicionar `DashboardProjectionService $projection` ao constructor.
   - **Antes** do `Cache::flexible(...)`:
     ```php
     if (
         (bool) config('board.dashboard.use_projection', false)
         && $ctx->cacheSegment() !== 'none'
         && $ctx->cacheSegment() !== 'global'
     ) {
         $valid = $this->projection->findValid((int) $ctx->tenantId(), $period);
         if ($valid !== null) {
             return $this->hydrateSharedFromPlain($valid->payload);
         }
     }
     ```
   - Fallback (L2 flexible) **intacto**.

## 7. Regras de segurança / tenancy

- `TenantDashboardSnapshot` usa `BelongsToTenant` — `TenantScope` aplica `tenant_id` automaticamente em queries de leitura via Eloquent. Operações administrativas do job/command usam `withoutGlobalScopes` explicitamente, mas filtram por `tenant_id` no query builder.
- `DashboardProjectionService::findValid` recebe `int $tenantId` **explícito** — nunca `auth()`/`request()`.
- `markStale` é o **único** método que faz UPDATE em projections de **outros** tenants; só é chamado via observer-trigger no contexto correcto. Validar em teste 13.
- `payload` JSON **não** contém `tenant_id` de OUTROS tenants (já vem dos providers que aplicam `restrictToTenant`).
- Cross-tenant test obrigatório: criar Task em A → markStale(A) é called; markStale(B) **não** é called.
- ON DELETE CASCADE: se o tenant for soft-deleted, a constraint não dispara (soft-delete só marca `deleted_at`); se for hard-deleted, projections vão junto. Comportamento aceite.
- Job runtime: tempo médio < 500 ms por tenant × período; se ultrapassar 5 s, fail (job `timeout = 30`).

## 8. Estratégia de chaves / IDs / nomes

### Tabela e índices

```sql
CREATE TABLE tenant_dashboard_snapshots (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    tenant_id BIGINT UNSIGNED NOT NULL,
    period VARCHAR(16) NOT NULL,
    payload JSON NOT NULL,
    is_stale BOOLEAN NOT NULL DEFAULT 0,
    refreshed_at TIMESTAMP NULL,
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL,
    UNIQUE KEY tenant_dashboard_snapshots_tenant_period_unique (tenant_id, period),
    KEY tenant_dashboard_snapshots_stale_refreshed_idx (is_stale, refreshed_at),
    CONSTRAINT tenant_dashboard_snapshots_tenant_id_foreign
        FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON DELETE CASCADE
);
```

### Shape do `payload`

```json
{
  "version": "v1",
  "hero": { /* HeroSummary::toArray() */ },
  "operations": { /* OperationsBlock::toArray() */ }
}
```

`version` deve igualar `config('board.dashboard.snapshot_version')` no momento do refresh; em leitura, projection com `payload.version` diferente do config actual é tratada como **stale** (fallback automático).

### Naming de classes

- Modelo: `App\Models\TenantDashboardSnapshot`.
- Service: `App\Services\Dashboard\Executive\Projection\DashboardProjectionService`.
- Job: `App\Jobs\Dashboard\RefreshTenantDashboardSnapshotJob`.
- Command: `App\Console\Commands\Dashboard\RefreshProjectionsCommand` → signature `dashboard:refresh-projections`.

9. Flag `use_projection=true` + projection com `refreshed_at = now-11min` → fallback L2.
10. Flag `use_projection=false` → projection **ignorada** mesmo válida; L2 flexível usado.

## 12. Rollback

1. `BGP_DASHBOARD_USE_PROJECTION=false` em produção.
2. `php artisan config:clear` + `php artisan cache:clear`.
3. Read service ignora completamente a projection; comportamento volta ao fluxo L2 + observabilidade (19B.1 + 19B.2).
4. Job continua a correr (sem dano — apenas escreve rows que ninguém lê).
5. Para desactivar totalmente: comentar entry em `Kernel::schedule`.
6. Para remover dados: `TRUNCATE tenant_dashboard_snapshots` (opcional, sem impacto funcional).

**Rollback completo** (último recurso, se a tabela introduzir problemas de performance/IO):

7. Drop migration: `php artisan migrate:rollback --step=1`.
8. Reverter código via revert do PR.
