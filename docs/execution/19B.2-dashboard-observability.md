# Fase 19B.2 — Observabilidade leve do cache do Executive Dashboard

> Especificação executável. O Executor deve seguir este documento sem precisar do chat.
> Origem arquitectural: [`docs/features/dashboard.md`](../features/dashboard.md) → roadmap "Próximos blocos 19B". Convenção `docs/execution/` em [`.cursor/rules/documentation.mdc`](../../.cursor/rules/documentation.mdc).
> **Reordenação proposta:** esta fase **substitui** a 19B.2 original (projection table), que passa a ser **19B.3+**. Razão: instrumentar antes de optimizar (`foundation-first.mdc`).

## 1. Contexto

A Fase 19B.1 fechou a invalidação síncrona dos caches **L1** (`DashboardMetricsService`) e **L2** (`ExecutiveDashboardReadService`) do dashboard executivo. **Não existe ainda instrumentação para responder a perguntas operacionais elementares**:

- O cache está realmente a ser servido (hit ratio)?
- Os TTLs (L1 90 s; L2 60 s stale / 120 s expire) são apropriados ao tráfego real?
- A invalidação por evento (19B.1) está a disparar com a frequência esperada (≪ 1 por leitura)?
- O `cacheSegment === 'global'` (D11) realmente se beneficia de TTL natural, ou super_admin está em miss constante?

Sem estas métricas, qualquer decisão futura (projection table, pre-warm, alteração de TTL) é especulativa. Esta fase fecha esse buraco de observabilidade **antes** de qualquer optimização adicional.

## 2. Decisões finais confirmadas

| Decisão | Conteúdo |
|---|---|
| **D13** — Observabilidade leve, **sem APM externo** | Instrumentação interna por counters em cache (`Cache::increment`) + leitura via comando Artisan. **Sem** dependências novas (Sentry, Datadog, OpenTelemetry, Prometheus). Equivalência a APM externo fica fora do âmbito; entra numa fase futura se houver justificativa. |
| **D14** — Granularidade temporal **diária** | Counters reset implícito por chave `{...}:{Y-m-d}`. TTL fixo de **7 dias** por contador. Suficiente para detectar variações dia-a-dia sem custo de armazenamento longo. |
| **D15** — Sem `tenant_id` em chaves de observability | Counters são **agregados globais** (`dashboard:obs:l1:hit:2026-05-12`). **Nunca** `dashboard:obs:l1:hit:t_42:...`. Privacidade primeiro: o ratio agregado responde a "está a funcionar?" sem expor padrão de uso por tenant. |
| **D15a** — Detecção hit/miss por `Cache::has()` | Estimativa aceite (race condition `has`→`remember` ±1 % off). Não justifica reescrever `DashboardMetricsService` para distinguir hit/miss de forma exacta. Hit ratio agregado é suficiente para decisões. |
| **D16** — Pontos de instrumentação **exactos** | Apenas **3 locais**: `DashboardMetricsService::getMetrics`, `ExecutiveDashboardReadService::loadOrComputeShared`, `ExecutiveDashboardCacheInvalidator::invalidateForTenant`. **Sem** instrumentação espalhada por providers, observers ou widgets. |
| **D17** — Stale-hit do `Cache::flexible` **fora do âmbito** | Detectar se o `flexible()` devolveu valor stale requer wrap interno do Laravel (não exposto). Documentar como gap; reabrir se necessário em fase futura. |
| **D18** — Leitura via **Artisan command** | `php artisan dashboard:cache-stats --day=2026-05-12 [--json]`. **Sem** widget Filament, sem endpoint API, sem persistência DB. Counters vivem só em cache (TTL 7 dias). |

## 6. Implementação passo a passo

> Padrão: **1 PR único** (escopo curto, baixo risco). `php artisan test` verde antes do merge.

### Passo 1 — Serviço de observability

1. Criar `App\Services\Dashboard\Executive\Observability\ExecutiveDashboardObservability`:
   - `final class`, constructor `__construct(private readonly Illuminate\Contracts\Cache\Repository $cache)`.
   - Constantes:
     ```php
     public const COUNTER_TTL_SECONDS = 60 * 60 * 24 * 7; // 7 dias
     public const PREFIX = 'dashboard:obs:';
     ```
   - Métodos públicos:
     - `recordL1Hit(): void` → `$this->increment('l1:hit')`
     - `recordL1Miss(): void` → `$this->increment('l1:miss')`
     - `recordL2Hit(): void` → `$this->increment('l2:hit')`
     - `recordL2Miss(): void` → `$this->increment('l2:miss')`
     - `recordInvalidation(): void` → `$this->increment('invalidations')`
     - `snapshotFor(\DateTimeInterface $day): array` — devolve estrutura §8.
   - Privado:
     - `increment(string $bucket): void` — constrói chave `{PREFIX}{bucket}:{Y-m-d}`, faz `$this->cache->add($key, 0, COUNTER_TTL_SECONDS)` (só inicializa se não existir) seguido de `$this->cache->increment($key)`.
   - **Sem** `auth()`, `request()`, `session()`. **Sem** `tenant_id`. **Sem** logging para ficheiro.

### Passo 2 — Instrumentação dos 3 locais (D16)

1. `DashboardMetricsService::getMetrics`:
   - Injectar `ExecutiveDashboardObservability $obs` no constructor (depois de qualquer dependência existente).
   - **Antes** de `Cache::remember($key, ...)`, chamar `Cache::has($key)`; se `true` → `$obs->recordL1Hit()`; senão → `$obs->recordL1Miss()`.
   - **Não** alterar a chamada a `Cache::remember` nem o comportamento de retorno. Race condition `has`→`remember` é aceitável (D15a).
2. `ExecutiveDashboardReadService::loadOrComputeShared`:
   - Injectar `ExecutiveDashboardObservability $obs` no constructor.
   - Skip se `$ctx->cacheSegment() === 'none'` (o early-return já existe; nenhuma instrumentação para segmento `none`).
   - Caso contrário, **antes** de `Cache::flexible($l2Key, …)`, chamar `Cache::has($l2Key)`; se `true` → `recordL2Hit()`; senão → `recordL2Miss()`.
3. `ExecutiveDashboardCacheInvalidator::invalidateForTenant`:
   - Injectar `ExecutiveDashboardObservability $obs` no constructor.
   - Após o loop de `forget`, chamar `$obs->recordInvalidation()` **uma vez por chamada** a `invalidateForTenant`. Não 1× por chave esquecida — agregamos por evento.

### Passo 3 — Comando Artisan

1. Criar `App\Console\Commands\Dashboard\CacheStatsCommand`:
   - Signature: `dashboard:cache-stats {--day= : YYYY-MM-DD (default: hoje, timezone do app)} {--json : output JSON ao invés de tabela}`.
   - `description = 'Estatísticas diárias do cache do dashboard executivo (L1, L2, invalidações).'`.
   - `handle(ExecutiveDashboardObservability $obs): int`:
     - Parse `--day` ou usa `CarbonImmutable::today()`. Falha se data inválida (`return self::FAILURE`).
     - `$snap = $obs->snapshotFor($day)`.
     - Se `--json`: `$this->line(json_encode($snap, JSON_PRETTY_PRINT | JSON_UNESCAPED_SLASHES))`.
     - Senão: `$this->table(['Métrica', 'Valor'], …)` com linhas para L1 hits/misses/ratio, L2 hits/misses/ratio, invalidations, day.
     - `return self::SUCCESS`.
2. Auto-discovery do Laravel encontra o comando em `app/Console/Commands/`.

### Passo 4 — Documentação

- `.cursor/rules/dashboard.mdc` — nova subsecção **"Observabilidade (19B.2)"** depois de "Invalidação de cache (19B.1)". Texto curto: D13–D18 + comando + chaves §8.

## 7. Regras de segurança / tenancy

- **Counters agregados globais por dia, nunca por tenant** (D15). Validado nos testes 6 e sentinela §12.5.
- **Sem PII em chaves/logs**: nenhum `tenant_id`, `user_id`, `email`, nome, ou payload de DTO em qualquer local de observability.
- **TTL fixo (7 dias)** — counters não persistem indefinidamente; rotação automática.
- **Comando Artisan apenas via CLI** — não exposto via HTTP nesta fase.
- **Sem dependências em `auth()` / `request()` / `session()`** no serviço de observability nem nos 3 locais instrumentados.
- **Driver de cache `array` em testes** garante isolamento entre cenários; `Cache::flush()` no `setUp`.

## 8. Estratégia de chaves / nomes

### Chaves de counter (exactas)

```
dashboard:obs:l1:hit:{Y-m-d}
dashboard:obs:l1:miss:{Y-m-d}
dashboard:obs:l2:hit:{Y-m-d}
dashboard:obs:l2:miss:{Y-m-d}
dashboard:obs:invalidations:{Y-m-d}
```

`{Y-m-d}` é a data **local do app** (timezone `config('app.timezone')`); evita inconsistência com `now()` usado em outras partes da Board.

### Shape de `snapshotFor($day)`

```json
{
  "day": "2026-05-12",
  "l1": { "hits": 1840, "misses": 22, "hit_ratio": 0.988 },
  "l2": { "hits": 1320, "misses": 19, "hit_ratio": 0.986 },
  "invalidations": 47
}
```

- `hit_ratio = hits / max(1, hits + misses)`, arredondado a 3 casas. **Nunca** divisão por zero.
- `day` em ISO-8601-date (sem hora). Estável para tooling externo (jq, scripts shell) no futuro.
- Estrutura **estável**: se evoluir (ex.: adicionar `stale_hits`), bumpar version no documento e adicionar campo opcional — nunca remover/renomear.

## 9. Mapas obrigatórios

### Pontos de instrumentação (exactos)

| Ficheiro | Método | Counter incrementado | Condição |
|---|---|---|---|
| `DashboardMetricsService` | `getMetrics` | `recordL1Hit` ou `recordL1Miss` | sempre (toda chamada conta) |
| `ExecutiveDashboardReadService` | `loadOrComputeShared` | `recordL2Hit` ou `recordL2Miss` | só se `cacheSegment !== 'none'` |
| `ExecutiveDashboardCacheInvalidator` | `invalidateForTenant` | `recordInvalidation` | 1 por chamada (não por chave) |

### Não instrumentar nesta fase

- Providers (`HeroProvider`, `KpiStripProvider`, `OperationsProvider`, `PrioritiesProvider`, `ActivityFeedProvider`).
- Observers (já contribuem indirectamente via invalidator).
- Page/widgets Filament.
- `Cache::remember` / `Cache::flexible` internos (race condition aceite — D15a).

## 10. Testes obrigatórios

> Runner: PHPUnit. Driver de cache: `array` (default de testes). `Cache::flush()` no `setUp` de cada teste.

### Unit — `ExecutiveDashboardObservability` (6 testes)

1. `recordL1Hit` incrementa chave `dashboard:obs:l1:hit:{hoje}` de 0 → 1 → 2 em chamadas sucessivas.
2. `recordL1Miss`, `recordL2Hit`, `recordL2Miss`, `recordInvalidation` análogos (sub-testes ou data provider).
3. `snapshotFor($day)` devolve shape canónico §8 com `hit_ratio` correcto a 3 casas decimais.
4. `snapshotFor($day)` com counters todos a zero devolve `hit_ratio: 0.0` (sem divisão por zero).
5. **TTL**: após `Carbon::setTestNow(now()->addDays(8))`, `Cache::get($key)` devolve `null` (counter expirou). Cobre rotação automática.
6. **D15 — anti-leak**: nenhuma chave criada pelo serviço contém substrings `t_`, `user_`, `email`, `@`, ou sequências de dígitos > 4. Asserção sobre `Cache::get` em chaves "negativas" (ex.: `dashboard:obs:l1:hit:t_42:2026-05-12`) — todas `null`.

### Feature — instrumentação L1/L2 (2 testes)

7. Após 1 leitura fresca via `DashboardMetricsService::getMetrics($user)` → counter `l1:miss` = 1, `l1:hit` = 0; segunda leitura imediata → `l1:miss` = 1, `l1:hit` = 1.
8. Após 1 leitura fresca via `ExecutiveDashboardReadService::read($user)` em tenant válido → `l2:miss` = 1, `l2:hit` = 0; segunda leitura → `l2:miss` = 1, `l2:hit` = 1. **Em segmento `none`** (`User` sem tenant que **não** seja `super_admin` — bloqueado pelo Gate, mas se for forçado em teste com construção directa, nenhum counter L2 é incrementado).

### Feature — invalidator (1 teste)

11. Após `Task::factory()->create(['tenant_id' => $tenant->id])` (dispara `TaskObserver::created` → `invalidateForTenant`), counter `invalidations` incrementa em **1** (não 6, apesar das 6 chaves esquecidas — D16: 1 invocação = 1 counter).

### Feature — comando Artisan (2 testes)

9. `php artisan dashboard:cache-stats --day=2026-05-12 --json` devolve JSON parseável com shape §8 e exit code 0. Pré-popular counters com `Cache::put` para valores conhecidos.
10. `php artisan dashboard:cache-stats` (sem `--day`) usa data de hoje, imprime tabela (`$this->table` na output) e exit code 0. `--day=banana` falha com exit code != 0.

### Critério de saída

- Total adicionado: **≈ 11 testes**, **≈ 30 assertions**.
- Suite total: **≥ 328/328 verde** (317 actuais após 19B.1 + ≈ 11 novos).
- Zero falhas em `php artisan test --filter='Api|Dashboard|Observer|Observability'`.
