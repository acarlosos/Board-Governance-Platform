# Fase 19B.4 — Endpoint API `GET /api/v1/dashboard/snapshot`

> Especificação executável. O Executor deve seguir este documento sem precisar do chat.
> Origem arquitectural: [`docs/features/dashboard.md`](../features/dashboard.md) → "Próximos blocos 19B" + [`docs/features/api.md`](../features/api.md). Convenção em [`.cursor/rules/documentation.mdc`](../../.cursor/rules/documentation.mdc).

## 1. Contexto

A API v1 (Fase 16/17/18) já expõe `auth`, `boards`, `meetings`, `tasks`, `notifications` com padrão estabelecido: Sanctum, ability check via `abilities:xxx` middleware, gate + `ApiResponder` envelope. A documentação OpenAPI vive em `docs/openapi/v1.yaml`.

O dashboard executivo (19A.3) foi desenhado com `ExecutiveDashboardSnapshot` **API-ready desde o dia 1** (D8: datetimes ISO8601, enums como `->value`, sem objects Eloquent). Esta fase **expõe** esse snapshot por HTTP, permitindo apps móveis/integradores ler o estado executivo sem render Filament.

## 2. Decisões finais confirmadas

| Decisão | Conteúdo |
|---|---|
| **D26** — Endpoint único | `GET /api/v1/dashboard/snapshot?period=this_month\|last_30_days\|all_time`. Sem `/api/v1/dashboard/{recurso}` adicional — a unidade é o snapshot completo. |
| **D27** — Ability **reaproveita `reports:read`** | **NÃO criar** `dashboard:read` novo. Razão: `view_reports → reports:read` já está mapeado em `features/api.md` §"Matriz abilities × permissões"; o dashboard executivo é semanticamente um "report". Evita duplicação no `CapabilitiesService` e seeder. |
| **D28** — Authorization dual (D8 vinculativo) | Middleware `abilities:reports:read` valida o token. Controller chama `Gate::forUser($actor)->authorize('view-executive-dashboard')` (gate já existe da 19A.6 — registado em `AuthServiceProvider`). Regra "permitido = Policy ∧ Ability" do `features/api.md` §"Matriz abilities × permissões" é mantida. |
| **D29** — Sem cache adicional no controller | `ExecutiveDashboardReadService` já cacheia (L1 + L2 + projection se 19B.3 ligada). Controller chama directamente. |
| **D30** — Resource API estável | `DashboardSnapshotApiResource` mapeia o DTO para JSON via `$dto->toArray()`. Datetimes ISO8601, enums `->value`, sem campos `null` desnecessários. **Não** acrescentar campos derivados no resource — o DTO já tem o que precisa. |
| **D31** — Rate limit | 60 req/min por user (alinhado com `GET /auth/me`, `GET /auth/tokens` em `features/api.md` §Rate limiting). Implementação via `RateLimiter::tooManyAttempts` como nos demais GETs. |
| **D32** — Sem endpoint write | Snapshot é **read-only** (D8). Nenhum POST/PATCH/DELETE em `/dashboard/*` nesta fase. |
| **D33** — Sem versão `v2` antecipada | Mantém-se `v1`. Mudança de shape do DTO requer bump em `config('board.dashboard.snapshot_version')` e novo endpoint `/api/v2/dashboard/snapshot` (não nesta fase). |

## 6. Implementação passo a passo

> Padrão: **1 PR único** — escopo pequeno, baixo risco. `php artisan test` verde antes do merge.

### Passo 1 — Request, Controller, Resource

1. `SnapshotIndexRequest`:
   - `authorize(): bool` → `true` (auth via middleware Sanctum + abilities; gate no controller).
   - `rules(): array` → `['period' => ['nullable', 'string', new \Illuminate\Validation\Rules\Enum(DashboardMetricsPeriod::class)]]`. Aceita `period` ausente → defaultar a `ThisMonth` no controller.
2. `DashboardController` (extends `App\Http\Controllers\Controller`):
   - Constructor: `__construct(private readonly ApiResponder $responder) {}`.
   - Método `snapshot(SnapshotIndexRequest $request, ExecutiveDashboardReadService $svc): JsonResponse`:
     ```php
     $actor = $request->user();
     if (! $actor instanceof User) {
         return $this->responder->unauthorized();
     }

     $key = 'api:v1:dashboard:snapshot:user:'.$actor->getKey();
     if (RateLimiter::tooManyAttempts($key, 60)) {
         return $this->responder->rateLimited();
     }
     RateLimiter::hit($key, 60);

     Gate::forUser($actor)->authorize('view-executive-dashboard');

     $period = $request->validated()['period'] ?? null;
     $periodEnum = $period
         ? DashboardMetricsPeriod::from((string) $period)
         : DashboardMetricsPeriod::ThisMonth;

     $snapshot = $svc->read($actor, $periodEnum);

     return $this->responder->ok(new DashboardSnapshotApiResource($snapshot), 200);
     ```
3. `DashboardSnapshotApiResource extends JsonResource`:
   - `toArray(Request $request): array` → `$this->resource->toArray()` (o DTO já implementa `toArray()` desde 19A.3). Não acrescentar nem omitir campos.

### Passo 2 — Rota

1. Em `routes/api.php`, dentro do grupo `auth:sanctum`:
   ```php
   Route::get('dashboard/snapshot', [DashboardController::class, 'snapshot'])
       ->middleware('abilities:reports:read')
       ->name('api.v1.dashboard.snapshot');
   ```
2. Confirmar ordem: depois de `notifications/*` (ordem alfabética não obrigatória, mas mantém consistência).

### Passo 3 — OpenAPI

1. Em `docs/openapi/v1.yaml`, adicionar sob `paths`:
   ```yaml
   /dashboard/snapshot:
     get:
       tags: [Dashboard]
       summary: Snapshot completo do dashboard executivo
       security:
         - bearerAuth: [reports:read]
       parameters:
         - in: query
           name: period
           schema:
             type: string
             enum: [this_month, last_30_days, all_time]
             default: this_month
       responses:
         '200':
           description: OK
           content:
             application/json:
               schema:
                 type: object
                 properties:
                   success: { type: boolean, example: true }
                   data: { $ref: '#/components/schemas/DashboardSnapshot' }
                   meta: { $ref: '#/components/schemas/Meta' }
         '401': { $ref: '#/components/responses/Unauthorized' }
         '403': { $ref: '#/components/responses/Forbidden' }
         '422': { $ref: '#/components/responses/Validation' }
         '429': { $ref: '#/components/responses/RateLimited' }
   ```
2. Adicionar schema `DashboardSnapshot` em `components.schemas`:
   ```yaml
   DashboardSnapshot:
     type: object
     required: [version, period, cache_segment, generated_at, hero, kpis, operations, priorities, activity]
     properties:
       version: { type: string, example: "v1" }
       period: { type: string, enum: [this_month, last_30_days, all_time] }
       cache_segment: { type: string, example: "t_42" }
       generated_at: { type: string, format: date-time }
       hero: { $ref: '#/components/schemas/HeroSummary' }
       kpis: { $ref: '#/components/schemas/KpiStrip' }
       operations: { $ref: '#/components/schemas/OperationsBlock' }

         type: array
         items: { $ref: '#/components/schemas/PriorityItem' }

         type: array
         items: { $ref: '#/components/schemas/ActivityItem' }
   ```
   Sub-schemas (`HeroSummary`, `KpiStrip`, `OperationsBlock`, `PriorityItem`, `ActivityItem`) — derivar dos DTOs `App\Services\Dashboard\Executive\Snapshot\*` (cada um já tem `toArray()` estável). Manter convenção `snake_case` no JSON.

## 7. Regras de segurança / tenancy

- **Tenancy via user autenticado** (regra cardinal `features/api.md` §"Tenancy v1"). Nenhum `?tenant_id=` aceito. `super_admin` continua a obter snapshot global (via `ReportingContext::fromUser`).
- **Dual check**: middleware `abilities:reports:read` falha → 403 (Sanctum responde). Gate `view-executive-dashboard` falha → 403 via `AuthorizationException`. Permitido = ambos passam.
- **Sem `find(id)` antes de autorizar** (regra `features/api.md`). Aqui não existe `id` — escopo é sempre o tenant do user.
- **User sem tenant + não super_admin** → gate `view-executive-dashboard` retorna `false` (comportamento 19A.6); responde 403.
- **Erro do read service nunca propaga payload bruto** — `ApiResponder` já envelopa em `error.code`/`error.message` sem stack trace.
- **Logs**: nenhum log custom no controller. `AuditLoggerService` não é chamado (read-only, leitura legítima não é auditável neste momento — pode entrar em fase futura se Compliance pedir).
- **Rate limit per-user (60/min)** previne abuso. Sem rate limit por IP nesta fase (consistente com `/auth/me`).

## 8. Estratégia de chaves / IDs / nomes

- Route name: `api.v1.dashboard.snapshot` (alinhado com `api.v1.tasks.index`, `api.v1.boards.index`).
- Rate limit key: `api:v1:dashboard:snapshot:user:{user_id}` (segue padrão `api:v1:{resource}:{action}:user:{id}`).
- Resource: `DashboardSnapshotApiResource` (sufixo `ApiResource` — convenção da Fase 16/17).
- Sem novas chaves de cache (read service já gere).

## 10. Testes obrigatórios

> Runner: PHPUnit. `RefreshDatabase` + `RolesAndPermissionsSeeder` no `setUp`.

### Feature — endpoint (8 testes)

1. `GET /dashboard/snapshot` sem bearer → **401** + envelope `success: false, error.code: unauthenticated`.
2. Bearer com token **sem** `reports:read` → **403** + `error.code: forbidden_ability`.
3. Bearer válido + user **com** `view_reports` + tenant ativo → **200** + shape:
   - `success: true`
   - `data.version`, `data.period`, `data.cache_segment`, `data.generated_at`, `data.hero`, `data.kpis`, `data.operations`, `data.priorities`, `data.activity`
   - `meta.request_id`, `meta.api_version: "v1"`
4. `?period=last_30_days` → `data.period == "last_30_days"`.
5. `?period=banana` → **422** com `error.fields.period` populado.
6. User **sem** `view_reports` mas com `reports:read` no token → **403** (gate falha).
7. User sem tenant + não super_admin → **403** (gate `view-executive-dashboard` falha).
8. Rate limit: 60 chamadas em sequência todas 200; 61ª → **429**.

### Feature — cross-tenant (2 testes)

9. User do tenant A consulta endpoint → `data.cache_segment == "t_{A.id}"`; estatísticas reflectem apenas dados de A.
10. Super_admin consulta endpoint → `data.cache_segment == "global"`; sem leakage do contexto do super_admin para um tenant arbitrário.
